---
title: "Webhooks"
description: "Receive signed VoucherPay events and use them to fulfill orders safely."
---

Webhooks are the durable confirmation path for payment and voucher state.

Do not rely only on success redirects or frontend callbacks to mark orders paid.

## Delivery format

VoucherPay sends JSON:

```json
{
  "id": "evt_...",
  "type": "payment.captured",
  "createdAt": "2026-05-12T12:00:00.000Z",
  "data": {
    "paymentSessionId": "cm...",
    "externalReference": "ORDER-1234",
    "voucherReference": "VP-..."
  }
}
```

Headers:

| Header | Meaning |
| --- | --- |
| `VoucherPay-Signature` | HMAC signature header. |
| `VoucherPay-Event-Type` | Event type, for example `payment.captured`. |
| `VoucherPay-Event-Id` | Event ID. |
| `VoucherPay-Delivery-Id` | Delivery attempt ID. |

## Signature format

`VoucherPay-Signature` has this format:

```text
t=<unix-seconds>,v1=<hex-hmac-sha256>
```

The signed payload is:

```text
<timestamp>.<raw request body>
```

The default tolerance is 5 minutes.

## Verify with the server SDK

```ts
import { VoucherPay } from "@voucherpay/server";

const voucherpay = new VoucherPay(process.env.VOUCHERPAY_SECRET_KEY!);

export async function POST(request: Request) {
  const rawBody = await request.text();
  const signature = request.headers.get("VoucherPay-Signature");

  const event = voucherpay.webhooks.constructEvent(
    rawBody,
    signature,
    process.env.VOUCHERPAY_WEBHOOK_SECRET!,
  );

  switch (event.type) {
    case "payment.captured":
      // Mark order paid. Use event.data.externalReference to find your order.
      break;
    case "payment.failed":
      // Mark failed or allow retry.
      break;
  }

  return Response.json({ received: true });
}
```

## Raw body requirement

Signature verification must use the exact raw request body. If your framework parses JSON before verification, configure the webhook route to expose the raw body.

## Retry behavior

VoucherPay retries failed deliveries on this schedule:

```text
1m -> 5m -> 30m -> 2h -> 6h -> 12h -> 24h -> 48h
```

Deliveries can be:

| Status | Meaning |
| --- | --- |
| `PENDING` | Waiting to be delivered. |
| `IN_PROGRESS` | Currently being delivered. |
| `DELIVERED` | Merchant endpoint returned success. |
| `DEAD` | Retry schedule exhausted. |

## Processing rules

- Return `2xx` only after durable processing.
- Store event IDs and ignore duplicates.
- Keep processing idempotent.
- Do not assume event ordering across independent objects.
- Acknowledge quickly and move slow work to your own queue.
- Reconcile by `paymentSessionId` or `externalReference` when in doubt.

## Fulfillment mapping

| Event | Merchant action |
| --- | --- |
| `payment.captured` | Mark order paid and release fulfillment. |
| `payment.failed` | Mark failed or allow retry. |
| `voucher.transferred` | Record value moved to merchant books, if subscribed. |
| `voucher.held` | Record value is held. |
| `voucher.released` | Record hold/refund release state. |
| `withdrawal.requested` | Track withdrawal request. |
| `withdrawal.completed` | Mark payout completed. |

Only depend on events enabled for your merchant account.
