---
title: "Checkout flow"
description: "The customer and system journey from cart to payment capture and voucher issuance."
---

The customer experience should stay simple:

1. Customer adds an item to cart.
2. Customer clicks checkout.
3. Store backend creates a VoucherPay checkout session.
4. Store frontend mounts the VoucherPay widget.
5. Customer reviews the order context.
6. Customer accepts terms if required.
7. Hosted card-entry form loads.
8. Customer enters card details.
9. 3DS runs when requested by the acquirer.
10. VoucherPay shows success after capture.
11. Customer returns to the merchant thank-you page.

## What happens server-side

| Step | System | Detail |
| --- | --- | --- |
| Create session | Merchant backend | Calls `POST /v1/public/checkout-sessions` with merchant API key. |
| Gate status | Browser widget | Reads `/status` using `X-VoucherPay-Client-Secret`. |
| Accept terms | Browser widget | Calls `/accept-terms` with the client secret. |
| Start payment | Browser widget | Calls `/start-payment` after gates clear. |
| Card entry | Acquirer | Hosted card surface collects card details and 3DS. |
| Capture | VoucherPay | Reconciles acquirer status and issues voucher. |
| Fulfillment | Merchant backend | Uses signed webhook events to mark the order paid. |

## Do not rely on redirects alone

Redirects help the customer return to your store, but they are not proof of payment. Your backend should fulfill orders after a signed webhook or a server-side reconciliation confirms final state.

## Current limitations

- Checkout creation currently accepts `EUR` only.
- Cardholder KYC is not live. If a merchant config enables it, payment start fails closed.
- The browser must never receive your merchant API key.
