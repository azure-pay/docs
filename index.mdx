VoucherPay lets merchants accept card payments through a hosted acquiring flow while VoucherPay handles checkout session state, terms acceptance, voucher issuance, settlement tracking, webhooks, and merchant payout workflows.

These docs are for engineering and operations teams integrating VoucherPay into a store.

## What VoucherPay does

1. Your backend creates a checkout session for a known merchant and store order.
2. Your frontend mounts the VoucherPay widget with a session ID and client secret.
3. VoucherPay shows order context, terms when required, and the hosted card-entry surface.
4. The acquirer handles card entry and 3DS.
5. VoucherPay reconciles capture, issues a voucher, sends signed webhooks, and updates the merchant portal.
6. Voucher value moves through merchant books, holds, reserves, and withdrawal workflows.

## What you build

Most merchants build four pieces:

| Piece | Runs where | Purpose |
| --- | --- | --- |
| Checkout-session route | Your backend | Creates VoucherPay sessions using your merchant API key. |
| Checkout page | Your frontend | Mounts the VoucherPay widget using `sessionId` and `clientSecret`. |
| Webhook endpoint | Your backend | Receives signed payment and voucher events. |
| Admin/operator workflow | Merchant portal | Manages API keys, webhooks, payment review, payouts, and withdrawals. |

## Current production surface

| Area | Current state |
| --- | --- |
| Checkout create endpoint | `POST /v1/public/checkout-sessions` |
| Browser status endpoint | `GET /v1/public/checkout-sessions/:id/status` |
| Terms acceptance | Implemented. |
| Cardholder KYC | Not live. If enabled, payment start fails closed. |
| Checkout currency | `EUR` only today. |
| Webhook signature header | `VoucherPay-Signature` |
| Widget package | `@azure-pay/checkout-widget` |
| Server SDK package | `@voucherpay/server` |
